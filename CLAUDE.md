# CLAUDE.md — TaskFlow Codebase Guide

## Project Overview

TaskFlow is a personal daily task manager implemented as a **single self-contained HTML file** (`taskflow.html`). It is a Progressive Web App (PWA) with no build step, no npm dependencies, and no external JavaScript libraries. All HTML, CSS, and JavaScript live in one file.

**Backend**: [Supabase](https://supabase.com/) (PostgreSQL + REST API). The user enters their Supabase project URL and anon key via an onboarding screen; these are persisted in `localStorage` under the key `tf_cfg`.

---

## Repository Structure

```
taskflow/
└── taskflow.html    # The entire application (HTML + CSS + JS, ~1,083 lines)
```

There is no build system, no `package.json`, no compiler, and no test suite. Editing `taskflow.html` directly is the development workflow.

## Architecture

### File Sections (in order)

1. **`<head>`** — Meta tags, Google Fonts imports (Lora + JetBrains Mono)
2. **`<style>`** — All CSS (~500 lines), including CSS custom properties, layout, components, and animations
3. **`<body>`**:
   - `#onboard` — Initial setup screen for Supabase credentials
   - `#mainApp` — Main application UI (header, tabs, filter bar, task list, add-bar)
   - Modal sheets: `#shSchedule`, `#shSync`, `#shSettings`, `#shEdit`
4. **`<script>`** — All application logic (~500 lines of vanilla JavaScript)

### Global State Objects

| Object | Purpose |
|--------|---------|
| `CFG` | Configuration — Supabase `url` and `key` loaded from localStorage |
| `S` | Application state — `tasks[]`, `templates[]`, `lastRollover` |
| `UI` | UI state — active `tab`, `filter`, `optDaily`, `optSchedule`, `snoozeTaskId`, `selectedCat`, `recurType` |
| `expanded` | `Set<string>` of task IDs whose action rows are currently expanded |

## Data Models

### Task Object

```javascript
{
  id: string,             // Generated by uid() — timestamp + random suffix
  title: string,
  date: string,           // YYYY-MM-DD (today's date when created)
  done: boolean,
  isDaily: boolean,       // True if created from a recurring template
  rolledOver: boolean,    // True if carried forward from a previous day
  snoozedTo: string|null, // YYYY-MM-DD target date if snoozed, else null
  category: string|null,  // 'work'|'personal'|'health'|'home'|'learning'|'finance'|null
  recurType: string,      // 'daily'|'weekdays'|'weekends'
  createdAt: string,      // ISO 8601 timestamp
  completedAt: string|null, // ISO 8601 timestamp or null
}
```

### Template Object (recurring task template)

```javascript
{
  id: string,
  title: string,
  recurType: string,  // 'daily'|'weekdays'|'weekends'
}
```

### Supabase Schema

The app expects a single table `tasks` with a single row (`id = 'main'`):

```sql
create table tasks (
  id text primary key,
  data jsonb,
  updated_at timestamptz default now()
);

insert into tasks (id, data) values ('main', '{"tasks":[],"templates":[],"lastRollover":null}');
```

The entire application state is serialized into the `data` JSONB column.

## Key Functions

| Function | Description |
|----------|-------------|
| `init()` | Entry point — checks localStorage for config, shows onboard or starts app |
| `startApp()` | Shows main UI, triggers `syncNow()`, starts rollover |
| `syncNow()` | Fetches data from Supabase (`sbGet`), merges with local state, saves back (`sbSet`) |
| `render()` | Re-renders the entire task list and filter chips based on current state |
| `renderTaskList()` | Generates HTML for task cards filtered by current tab and category |
| `autosave()` | Debounced (800ms) write to Supabase — call after any state mutation |
| `rollover()` | Runs once per day — carries forward incomplete tasks, creates daily tasks from templates |
| `sbGet()` | Fetches `data` column from Supabase REST API |
| `sbSet(data)` | PATCHes `data` column to Supabase REST API |
| `exportData()` | Serializes `S` to the JSON shape stored in Supabase |
| `uid()` | Generates a unique ID (timestamp + random suffix) |
| `esc(str)` | HTML-escapes a string — **always use this when rendering user content** |
| `setSyncStatus(type, label)` | Updates the sync pill UI — types: `'ok'`, `'pending'`, `'error'` |
| `shouldShowToday(template)` | Returns true if a recurring template should produce a task today |

## State Management Pattern

All state mutations follow this pattern:

```javascript
// 1. Mutate the global state object S
S.tasks.push(newTask);
// or
S.tasks = S.tasks.map(t => t.id === id ? { ...t, done: true } : t);

// 2. Persist (debounced)
autosave();

// 3. Re-render UI
render();
```

Never mutate state without calling `autosave()` afterward. Never call `sbSet()` directly from UI handlers — always go through `autosave()` so writes are debounced.

## JavaScript Conventions

- **Naming**: camelCase for functions and variables. Short abbreviations for global objects (`CFG`, `S`, `UI`, `sb*`).
- **No frameworks**: Pure DOM APIs — `querySelector`, `innerHTML`, `classList`, `addEventListener`.
- **XSS prevention**: Always wrap user-provided strings in `esc()` before inserting into `innerHTML`.
- **Date format**: `YYYY-MM-DD` strings throughout. The constant `TODAY` holds today's date as a string.
- **Template literals**: Used for HTML generation in `render()` and `renderTaskList()`.
- **Event delegation**: Modal overlays use a single click handler on the sheet element.

## Supabase API Pattern

All Supabase calls use `fetch` with these headers:

```javascript
{
  apikey: CFG.key,
  Authorization: `Bearer ${CFG.key}`,
  'Content-Type': 'application/json'
}
```

- **GET**: `${CFG.url}/rest/v1/tasks?id=eq.main&select=data`
- **PATCH**: `${CFG.url}/rest/v1/tasks?id=eq.main` with JSON body `{ data: ..., updated_at: new Date().toISOString() }`

## Development Workflow

1. **Edit** `taskflow.html` directly in any text editor.
2. **Test** by opening the file in a browser (`file://` protocol works).
3. On first open, enter your Supabase project URL and anon key in the onboarding screen.
4. The app saves config to `localStorage`; subsequent opens skip onboarding.
5. To reset, run `localStorage.removeItem('tf_cfg')` in the browser console.

**There is no build step, no linter, and no automated test suite.** Verify changes manually in the browser.

## Security Notes

- **Always use `esc(str)`** when rendering any user-provided content into `innerHTML` to prevent XSS.
- Supabase credentials are stored in `localStorage` — treat them as sensitive. Do not log them.
- The app uses a single shared Supabase row (`id='main'`); there is no per-user access control. Row Level Security (RLS) should be configured on the Supabase side if multi-user isolation is needed.

## Recurrence Types Reference

| Value | Behavior |
|-------|---------|
| `daily` | Task created every day |
| `weekdays` | Task created Monday–Friday only |
| `weekends` | Task created Saturday–Sunday only |
